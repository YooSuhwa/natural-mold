## 핵심 결정사항 (사용자 승인)
- HttpOnly Cookie + CSRF body token (not Bearer/localStorage)
- JWT HS256, Access 1h, Refresh 30d (DB whitelist via RefreshToken 모델)
- bcrypt (passlib)
- super_user 단일 boolean 플래그 (RBAC 미도입)
- User-only tenancy + Workspace 확장 hook은 service layer 추상화로
- Email + Password만 (Google OAuth는 Phase 2)
- System credentials는 super_user 전용
- 첫 가입자는 자동 super_user

## Codebase Patterns (분석 결과)
- FastAPI + SQLAlchemy(async) + Alembic
- 대부분 모델에 user_id FK 존재 (Agent, BuilderSession, Trigger, Tool, Credential, DailySpendUser)
- conversation은 직접 user_id 없음 → Agent.user_id로 간접 검증 (chat_service.get_owned_conversation)
- enumeration oracle 방지 패턴: get_owned_*가 None 반환 → 라우터에서 404
- 시스템 도구는 user_id IS NULL → list 시 OR 필터로 포함
- Cipher V2 (credential_service)로 토큰 암호화 가능
- LangGraph checkpoint는 thread_id=conversation_id, user_id 컬럼 없음

## API 계약 (백 → 프 인터페이스)
- POST /api/auth/register {email, password, name} → {user, csrf_token} + Set-Cookie
- POST /api/auth/login {email, password} → {user, csrf_token} + Set-Cookie
- POST /api/auth/logout → {ok} + Clear-Cookie
- POST /api/auth/refresh → {csrf_token} + Set-Cookie
- GET /api/auth/me → {user: {id, email, name, is_super_user, created_at}}
- 모든 mutation에 X-CSRF-Token 헤더 필수 (auth/login, auth/register, auth/refresh 제외)
- 401 응답 시 프론트는 1회 /refresh 시도 → 실패 시 /login으로 redirect
- credentials: 'include' 항상 (HttpOnly cookie 자동 첨부)

## ADR-016 작성 중 발견한 결정 패턴 (피차이)
- **CHECK constraint로 `is_system` 일관성 강제**: `(is_system=FALSE OR user_id IS NULL)` — 코드 버그로 system 리소스에 user_id가 들어가는 사고를 DB 레벨에서 차단. tools/credentials 양쪽 동일 적용. 젠슨이 m22 마이그레이션에 ALTER TABLE ADD CONSTRAINT로 박을 것.
- **refresh replay 방어 = user 전체 폐기**: rotation 시 이전 row revoke만으로는 부족. 이미 revoked인 token 재사용 감지 시 해당 user의 active refresh를 일괄 revoke해서 force re-login. 단순 401만 반환하면 공격자가 다른 토큰으로 계속 시도 가능.
- **첫 가입자 super_user는 production에서 위험**: `ALLOW_FIRST_USER_AS_ADMIN=true` 환경변수로 토글 가능하게 하고, .env.example에 "운영자 가입 직후 false로 변경" 경고 명시. 베조스가 마이그레이션 가이드에 반영, 젠슨이 .env.example에 주석 추가.

## S3 → S5/S6/S7 인계
- (S5 저커버그) 모든 mutation에 `X-CSRF-Token` 헤더 필수. 예외는 `/api/auth/login`, `/register`, `/refresh`. `lib/api/client.ts`에서 mutation 메서드(POST/PATCH/PUT/DELETE)에 자동 첨부 + `credentials:'include'` 필수.
- (S5) system credentials 라우터 (`/api/system-credentials/*`)는 super_user 응답 외에 403 `forbidden` 반환. 일반 user에게 메뉴 자체를 숨기는 게 UX 좋음 (`user.is_super_user` 체크).
- (S6 젠슨 AI runtime) `system_credential_resolver.resolve_system_api_key`는 그대로 유지 — assistant_agent/image_gen는 super_user 콜러가 아니라 시스템 빌링 플로우. 이 정책 결정은 별도 이슈로 추적 필요 (일반 user의 builder 사용 시 누가 빌링하나? 현재는 system credential이 fallback되어 운영자 부담).
- (S6) 신규 마이그레이션 m37 — `credentials.user_id` nullable + CHECK constraint(`(is_system AND user_id IS NULL) OR (NOT is_system AND user_id IS NOT NULL)`)이 m36에 이미 있다고 가정. m36 검증 후 차이 있으면 m37로 보완. 데이터 백필(`UPDATE credentials SET user_id=NULL WHERE is_system=true`)도 m36에 포함되었어야 함 (S2 lessons에 기록된 PostgreSQL CheckViolation 회피).
- (S7 베조스 통합) 격리 매트릭스 테스트 시:
  1. user A의 credential id를 user B가 GET → 404 (enumeration oracle 안전)
  2. 일반 user가 `/api/system-credentials` GET → 403
  3. 일반 user가 `/api/models` POST → 403
  4. CSRF 헤더 누락 mutation → 403 `csrf_mismatch`
  5. cookie만 있고 X-CSRF-Token 없는 cross-site 시도 → 403
  6. user A 토큰으로 user B의 conversation에 mutation (update/delete/switch_branch) → 404

## 디자인/구현 결정 (저커버그 — 프론트엔드 S5)

- **Next.js 16 컨벤션**: `middleware.ts`는 deprecated. Next 16부터 `proxy.ts` + `export function proxy()` 로 명명. `node_modules/next/dist/docs/01-app/01-getting-started/16-proxy.md` 참조. 동작은 동일.
- **(auth) route group 노출**: Next.js의 route group은 URL에 영향을 주지 않으므로 `/login`, `/register` 같은 절대 경로만 보호 매트릭스에 추가. proxy의 PUBLIC_ROUTES에 명시적 등록 필요.
- **Bare layout 분기**: `AppLayout` (client component) 가 `usePathname()` 으로 `/login` `/register`를 감지해 사이드바 없이 children만 렌더. (auth)/layout.tsx는 그 안에서 2컬럼 그리드로 폼 카드를 배치.
- **CSRF 메모리 + 쿠키 fallback**: `lib/auth/csrf.ts` 에서 in-memory가 primary, `moldy_csrf` 쿠키가 fallback (페이지 새로고침 후 SPA 부팅 시). login/register/refresh 응답 body의 `csrf_token`을 `useAuth`가 `setCsrfToken()`으로 저장.
- **401 dedup**: `apiFetch`가 401 받으면 `tryRefresh()` 단일 promise로 모든 동시 요청 동기화. 성공 시 원 요청 1회 재시도, 실패 시 한 번만 fireSessionExpired() (1초간 dedup) → toast + queryClient.clear() + `/login?callbackUrl=…` redirect (current path 보존, // 시작은 거부).
- **Auth endpoint 화이트리스트**: `/api/auth/login`, `/api/auth/register`, `/api/auth/refresh` 는 CSRF 헤더 자동 첨부 + 401 refresh 흐름에서 제외. `/api/auth/logout`은 CSRF 필요 (백엔드 verify_csrf 의존).
- **OnboardingDialog 패턴**: React 19 `react-hooks/set-state-in-effect` 룰 회피 위해 outer는 props만 보고 mount 결정, inner는 `key={user.id}` + `useMemo(() => shouldShow(user), [user])` 로 초기 open 계산. 효과 안에서 setState 안 함.
- **Super user welcome toast**: 별도 훅 `use-super-user-welcome-toast.ts` — 효과 안에서 sessionStorage flag + toast.success 만 호출 (state sync 없음). 5분 이내 created_at이 fresh 조건.
- **shadcn 미설치 컴포넌트**: Alert/Avatar/Form/Label은 프로젝트에 없음 → AuthAlert는 lucide + 토큰 기반 inline 컴포넌트로 대체. Avatar는 initials 함수로 div 생성. Form/Label도 native `<label>` + Input 직접 조합 (react-hook-form/zod 미설치 — 의존성 추가 회피).
- **PasswordStrengthMeter 점수 규칙**: 길이 기반 단순 휴리스틱 (8/10/14/18자) + 영숫자기호 혼합 보너스. zxcvbn 도입은 나중에 교체 가능.
- **session-expired handler 등록 패턴**: `lib/api/client.ts` 의 setSessionExpiredHandler(handler)는 React 컨텍스트가 아니라 모듈 레벨 변수. `QueryProvider` 가 useEffect에서 등록/해제. 이렇게 하면 fetch 레이어가 React 의존성 없이 토스트/리다이렉트 트리거 가능.
- **Suspense 필수**: Next.js 16에서 `useSearchParams()`는 page에서 직접 호출 시 prerender error. `<Suspense fallback={null}>` 로 감싸야 함 (login/page.tsx).

- **cleanup_user_resources 동작**: (1) `Conversation.agent_id → Agent.user_id` JOIN 으로 thread_id 수집 → `agent_runtime.checkpointer.delete_thread(str(conv_id))` 반복 호출. checkpointer 가 raise 해도 try/except 로 cleanup 자체는 계속 — 사용자 삭제가 checkpoint store 일시 장애에 막히지 않음. (2) `UPDATE refresh_tokens SET revoked_at=now() WHERE user_id=? AND revoked_at IS NULL` 일괄 revoke (이미 revoked 된 row 는 forensic timestamp 보존). FK CASCADE 가 어차피 row 자체를 지우지만 cleanup 시점에 revoke 해두면 race window 에서도 토큰 무효화.
- **delete_user 동작**: `cleanup_user_resources` 호출 후 `db.delete(user)` + `flush`. commit 은 caller 책임. m36 의 ON DELETE CASCADE 로 agents/builder_sessions/agent_triggers/refresh_tokens/credentials(non-system)/daily_spend_user 자동 정리.
- **migrate_mock_to_real_user.py 사용법**: `uv run python scripts/migrate_mock_to_real_user.py --target-user-id <UUID> [--source-user-id <UUID>] [--dry-run] [--delete-source]`. source 기본값 `00000000-0000-0000-0000-000000000001` (레거시 PoC mock UUID). 6개 테이블 UPDATE(`agents`,`builder_sessions`,`agent_triggers`,`tools` non-system,`credentials` non-system,`daily_spend_user`) → 단일 트랜잭션 commit/rollback. dry-run 시 rollback. `--delete-source` 지정 시 마지막에 source User row DELETE(나머지는 CASCADE). idempotent — 두 번 실행해도 두번째는 0 row.
- **테스트 인프라 발견**: SQLite aiosqlite 는 `DateTime(timezone=True)` 컬럼 round-trip 시 tzinfo 를 떨어뜨림 → naive datetime 으로 비교해야 함. `assert dt1 == dt2` 직접 비교 금지, `replace(tzinfo=None)` 후 비교.
- **사이드 이펙트 없음 검증**: test_executor / test_hitl_middleware / test_model_fallback / test_user_cleanup / test_chat_service / test_chat_integration 88개 PASS. 전체 suite 의 149개 실패는 S3 에서 추가된 CSRF 가드 영향(test_thread_branch, test_tools, test_triggers, test_uploads_router, test_skills) — S6 작업과 무관, S5 인계 사항으로 이미 인지됨.

## 트랜잭션 경계 + commit 강제 (jensen S7 escalation fix)
- **패턴**: 라우터/서비스에서 카운터/상태 갱신(UPDATE) 후 raise 전에 `await db.commit()` 필수. FastAPI 의 `get_db` 의존성은 raise 시 자동 rollback 하므로 flush 만 한 변경은 영속되지 않음.
- **재현**: `authenticate` 실패 시 `record_login_failure` 만 호출하고 raise → `failed_login_attempts` 영구 0 → 5회 lockout 무력화. `rotate_refresh` replay 분기 `_revoke_all_active` UPDATE 후 raise → mass-revoke 롤백 → 도난 refresh 방어 무력화.
- **수정 위치**: `app/services/auth_service.py::authenticate` (record_login_failure 직후) / `rotate_refresh` (replay UPDATE 직후). 정상 경로의 commit 은 router 가 책임지므로 그대로 둠.
- **회귀 가드**: `tests/test_auth_login.py::test_wrong_password_returns_401_and_increments_counter`, `test_five_failures_lock_account_returns_423`, `tests/test_auth_refresh.py::test_refresh_replay_revokes_all_active`. xfail strict 제거.
- **유사 클래스 점검 대상**: 다른 라우터/서비스에서 카운터/감사 row UPDATE 후 raise 하는 코드 — 발견 시 동일 fix 필요.
