- `services/spend_writer.py` — `SpendEntry` dataclass + `DailySpendUpdateQueue` (asyncio.Queue + 백그라운드 drain task). 핵심 결정:
  - 단일 큐로 3개 axis 모두 처리 (분리하면 producer가 3번 enqueue 해야 함)
  - flush 트리거: `flush_interval_sec=5.0` 또는 `batch_size=100` 도달 시
  - drain: `wait_for(queue.get(), timeout=interval)` → greedy `get_nowait` until batch_size → `_flush_batch`
  - `_flush_batch`: entries를 axis별 `(date, target_id)`로 group → 각 테이블에 dialect-aware `INSERT ... ON CONFLICT (date, target_id) DO UPDATE SET total_X = X + EXCLUDED.X` (PG/SQLite). 그 외 dialect는 application-level upsert fallback.
  - producer (`add()`) 비차단: `put_nowait` (QueueFull / RuntimeError 시 warning + drop, agent 실행 막지 않음).
  - `stop()` graceful: 큐 잔여 flush → task cancel → await (with `contextlib.suppress`).
  - 모듈 전역 `spend_queue` 싱글턴.
- `hooks/builtin/spend_hook.py` — `agent_invoke` post hook에서 `spend_queue.add(SpendEntry(...))`. 0 token / 0 cost row는 skip (interrupt 등 비정상 종료 가드). failure isolation은 HookRegistry + 자체 try/except 이중 가드.

## Phase 1 → Phase 2 핵심 결정 (ADR-012 §3, §4)

### ResumeRequest dual-shape (transition)
```python
class Decision(BaseModel):
    type: Literal["approve", "edit", "reject", "respond"]
    edited_action: dict | None = None  # type=edit
    message: str | None = None         # type=reject | respond

class ResumeRequest(BaseModel):
    decisions: list[Decision] | None = None  # 표준 (신규)
    response: str | list[str] | dict | None = None  # @deprecated, 단일 respond decision으로 변환
    # 둘 중 하나 이상 필수 — model_validator로 검증
```

### INTERRUPT event payload dual emit
```typescript
// 표준 (신규)
{ event: 'interrupt', data: {
  action_requests: [{ name, args, description? }],
  review_configs: [{ action_name, allowed_decisions: ('approve'|'edit'|'reject'|'respond')[] }]
}}
// legacy (보존, transition window 동안 함께 emit)
{ event: 'interrupt', data: { interrupt_id, value: { type, question?, options?, message? } } }
```
streaming.py는 한 task의 한 interrupt당 두 SSE chunk를 발행한다 (표준 + legacy).
frontend는 표준 우선 처리, legacy fallback. 두 chunk가 같은 interrupt를 가리키므로
중복 처리 방지 로직 필요 (e.g. 표준 chunk 받으면 legacy chunk 무시 — 같은 interrupt_id 기준).

### Multi-action 큐 (표준만)
- 표준 `action_requests`는 길이 ≥ 1. 길이 1이면 단일 액션 = 기존 동작과 동치.
- 길이 ≥ 2이면 frontend가 배열 큐로 렌더 (Phase 2에서는 순차 카드 + 일괄 확정 버튼으로 단순 시작).

### 트리거 모드
- Phase 1에서 이미 `include_ask_user=False`일 때 `interrupt_on=None`으로 차단. 표준 미들웨어 인스턴스 자체가 주입되지 않음. **Phase 2에서 트리거 경로 추가 변경 없음.**

## Codebase Patterns (Phase 2 컨텍스트)

- Backend SSE: `event_names.INTERRUPT` 상수 재사용. dual emit 시 두 chunk가 같은 event 이름을 사용.
- Backend 테스트: pytest + aiosqlite in-memory. langchain internal 속성 fallback 의존 시 주석 명시.
- Frontend types: `SSEEvent`는 discriminated union — interrupt variant의 data를 union으로 확장 시 타입 가드 필요. `'action_requests' in data` in-operator로 분기.
- Frontend SSE: `streamSSEPost`로 일관 송신. `consumeStream`이 `case 'interrupt'`로 분기.
- Frontend HiTL: `HiTLContext`(`hitl-context.ts`)가 `onResume` 콜백 제공. `useHiTL()`로 tool UI에서 호출.
- Frontend i18n: `messages/ko.json`. 키 네이밍 `chat.approval.*`.

## Gotchas (Phase 2)

- `streaming.py:343` 의 `agent.aget_state(config)`는 generator finally 외부에서 실행. interrupt 발생 후 emit. **dual emit 시 한 task의 한 interrupt당 정확히 두 chunk(표준+legacy)만 발행** — 중복 주의.
- `was_interrupted=True` fallback 분기에도 표준 형식을 함께 emit해야 일관성 유지.
- Frontend `consumeStream` 의 `case 'interrupt'`: 현재 단일 `onInterrupt(payload)` 호출. 표준 payload 처리 분기 추가 시 type narrowing 필요.
- `lastInterruptIdRef.current = intrId` (stale 검증용)는 legacy `interrupt_id`에서 유래. 표준 payload 우선 처리 시 stale 검증이 끊기지 않도록, **legacy chunk가 함께 도착하면 그쪽에서만 갱신**하는 방식이 안전. 또는 표준 payload에 corrleation id를 함께 emit (서버 측에서 동일 interrupt_id를 두 chunk 모두에 포함).
- Phase 2 dual emit 권장 구현: 표준 chunk에도 `interrupt_id` 동봉 (frontend stale 검증/중복 dedupe에 활용). ADR-012 §4에는 명시되지 않았지만 transition window의 안전망으로 채택. 피차이가 wire-contract에서 명시 결정.
