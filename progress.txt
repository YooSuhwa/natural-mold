## Codebase Patterns (젠슨 M2 Slice A, 2026-05-18)
- Alembic 마이그레이션 SQLite 호환: `sa.Uuid()` 사용 (PG/SQLite 둘 다 호환). `op.create_index(..., postgresql_where=...)`는 PG에서만 partial index 생성; SQLite 테스트는 ORM이 `Base.metadata.create_all` 경로라 영향 없음.
- 순환 FK (`marketplace_items.latest_version_id` ↔ `marketplace_versions.id`): m40에서 (1) items create(FK 없이) → (2) versions create → (3) ALTER ADD FK. ORM은 `mapped_column(ForeignKey(..., use_alter=True, name=...))` + `relationship(..., post_update=True)`. `use_alter=True` 없이는 SQLite create_all에서 SAWarning("Can't sort tables for DROP")이 나오므로 ORM 쪽에도 use_alter 명시 필요.
- 예약어 `metadata`: 컬럼명은 `metadata`로 두되 attribute는 `metadata_json: Mapped[dict | None] = mapped_column("metadata", JSON, ...)` 패턴으로 분리.
- ruff UP037: `from __future__ import annotations` 사용 시 forward ref 모두 unquoted. 이미 자동 fix됨 — TYPE_CHECKING import도 unquoted로 변환 가능.
- SQLAlchemy 2.0 `selectinload(MarketplaceItem.latest_version)` 사용 시, latest_version_id가 NULL이면 lazy fallback도 필요 (None 체크 후 db.get).
- 라우터 mount 검증: `uv run python -c "from app.main import create_app; app = create_app(); [r.path for r in app.routes if 'marketplace' in r.path]"` — import error 없으면 mount 성공.
- SkillResponse origin/publication embed: ORM → Pydantic 변환 후 추가 필드 `response.origin_summary = ...` 등 직접 할당. `model_config = {"from_attributes": True}`는 ORM 필드만 매핑하고 우리 모듈 필드는 그대로 둠.
- 첫 단계 Alembic verify는 PostgreSQL 필요 (이미 `natural-mold-postgres-1` Up). docker 미실행 시 `docker-compose up -d postgres`.
- `_serialize_skill` 헬퍼 위치: router 모듈 안, `router = APIRouter(...)` 선언 이후. ruff isort 가끔 router 선언을 import 블록 뒤로 이동시키니 함수 정의 전에 router 선언이 오게 명시적으로 유지.

## Bezos Test Patterns (M2 Slice A 회귀, 2026-05-19)
- 기본 `client` fixture는 super_user 강제. 일반 사용자 시나리오는 `_client_for_user(CurrentUser)` 헬퍼로 별도 app + dependency_overrides 구성 필요 (패턴은 `tests/test_marketplace_access.py:_client_for_user`).
- enumeration oracle 검증: 비공개 item과 존재하지 않는 item이 같은 JSON 응답을 반환해야 함. `assert r_hidden.json() == r_missing.json()`로 envelope shape 동등성까지 확인 (status code만 보면 응답 키 차이로 oracle이 생길 수 있음).
- 접근 술어(`marketplace.access`) 단위 테스트는 ORM 인스턴스를 메모리에서 직접 구성 (`item.acl_entries = [...]`) — DB 왕복 불필요. 라우터 통합 테스트는 실제 DB seed + ASGI client.
- m41 backfill SQL은 동일 statement를 SQLite create_all 스키마에서도 실행 가능 — 별도 alembic 통합 테스트 없이도 backfill 의미를 가드할 수 있음 (`tests/test_skills_regression.py::test_m41_backfill_marks_package_skills_imported_by_me`).
- 공개 catalog 검색 노출 게이트는 **filter mechanic**(`?is_listed=true`)으로만 강제됨. 기본 `GET /items` 자체는 모든 visible 항목 노출. 프론트엔드가 catalog 뷰에서 `is_listed=true`를 보내는 책임. 향후 서버사이드 기본값 변경 시 `tests/test_marketplace_listing.py` 두 테스트 동시 업데이트. **[M2.5 update 2026-05-19]** 젠슨 정정 후 default `GET /items`은 owner+system+(public+listed)+restricted-ACL만 노출. `visibility=unlisted`은 list 항상 제외 + detail은 직접 ID 접근 허용. `?is_listed=true`는 explicit narrow (visibility scope과 AND). `?is_listed=false`는 moderation 큐 — super_user에게 미승인 public 노출, 일반 사용자에게는 자기 소유 unlisted까지만.

## Bezos Test Patterns (M2.5 listing 정정, 2026-05-19)
- 기본 `GET /items` default 동작이 Spec §10.1 일치: 일반 사용자는 owner+system+listed-public+restricted-ACL만, super_user는 전체.
- `?is_listed=true`와 default는 **다른 결과** — explicit filter는 visibility scope와 intersect하므로 owner items 중 is_listed=False인 것이 빠짐. test에서 둘을 동일하게 가정하면 fail.
- `?is_listed=false` 결과 set 크기는 user role 의존: super_user는 모든 미승인 public, regular user는 자기 owned 미승인만 (owner clause 우회).
- `visibility=unlisted` 필터를 명시해도 list 응답에 포함되지 않음 — service `_base_catalog_query`의 visibility_clauses에 unlisted가 없기 때문. detail 접근은 `can_view_item` 통과로 별개 경로.
- access test는 predicate(`can_view_item`)/detail(`GET /items/{id}`) 레이어에만 의존하므로 list 동작 변경 영향 없음. predicate가 `unlisted+published`를 True로 반환하는 것은 detail-link sharing 의도와 정합.
- Agent 생성자는 `slug`/`tools_config`/`assistant_enabled` 인자 받지 않음. 테스트에서 Agent 직접 생성 시 `id/user_id/name/description/system_prompt/model_id`만 사용.

## Codebase Patterns (피차이 추가)
- ORM 임포트 헤더 표준화: `Mapped` + `mapped_column` + `PG_UUID(as_uuid=True)` + `relationship` (SA 2.0 declarative). ADR-016 패턴과 동일.
- Circular FK 해결책: `marketplace_items.latest_version_id` ↔ `marketplace_versions.id`는 m40 안에서 (1) items FK 없이 생성 (2) versions 생성 (3) ALTER TABLE ADD CONSTRAINT. ORM은 `relationship(..., post_update=True)`.
- Partial UNIQUE INDEX 패턴: PostgreSQL `WHERE` 절을 `Index(..., postgresql_where=text("..."))`로 표현. system slug + owner slug 두 개.
- `metadata`는 SQLAlchemy 예약어 — 컬럼명은 `metadata`로 두되 attribute는 `metadata_json` 권장.
- `ON DELETE RESTRICT` 두 곳: `marketplace_installations.version_id`, `skill_credential_bindings.credential_id`. 나머지는 CASCADE 또는 SET NULL.
- 모듈 임포트 단방향 규칙: schemas/access/secret_scan/redaction은 leaf. service들은 cross-import 금지. agent_runtime은 redaction + credential_requirements만 import.

- `CredentialDefinition`은 `key`(고유), `display_name`, `category`, `properties: list[FieldDef]`만 있으면 등록 가능. `authenticate`/`test`는 선택 — marketplace용 8개는 모두 인터폴레이션/connectivity test 없는 raw 저장만.
- `FieldKind.PASSWORD` + `type_options={"password": True}`는 frontend mask hint. 일반 STRING(`username`/`access_key`/`base_url`)은 평문 입력 UI.
- `optional credential` 의미는 두 층에 있음: (a) `FieldDef.required=False` — 필드 자체 선택, (b) skill 요구사항 `CredentialRequirement.required=False` — 바인딩 자체 선택. `coupang_partners`는 (a)는 둘 다 required, (b)에서 `required=False`로 publish됨 — 정의 자체는 두 필드 다 필수.
- `validate_binding`은 5개 거부 룰을 한 함수에 모음: 소유권→requirement_key→credential 존재+소유→system 거부→definition_key 일치. 순서 중요 — enumeration safety를 위해 ownership을 가장 먼저.
- `Cross-user credential`은 `CREDENTIAL_NOT_FOUND` 404로 collapse (다른 user의 cred 존재 여부 leak 방지). `system credential` + `definition_key mismatch` + `unknown requirement_key`는 422 `MARKETPLACE_CREDENTIAL_MISMATCH` (사용자 본인의 의도 오류 — 구체적 사유 노출 OK).
- DELETE 바인딩은 idempotent 204 — 존재 여부와 무관 (rules/security.md, enumeration channel 방지). `await db.commit()`는 행 삭제 시에만 호출.
- 테스트 fixture에서 `client` fixture를 쓰면 super_user 강제 + CSRF 우회. `db: AsyncSession` fixture로 직접 ORM 행 생성. `TEST_USER_ID` 유저는 별도 `_make_test_user(db)` 헬퍼로 생성 (skill/credential의 FK 만족).
- `installation.status` derivation: DB의 `install_status='active'`이고 `missing_required_keys`가 비어있지 않으면 응답에서 `'needs_setup'`로 승급. `disabled`/`uninstalled`는 명시적 상태이므로 유지.

  - `new_copy`: 항상 신규 — 동일 item을 여러 번 설치 가능

- `update_installation`: dirty 시 strategy 강제. `overwrite`는 install_new_copy를 호출 후 installation row의 `installed_skill_id`만 새 row로 reparent + 중복 row 삭제. URL/외부 참조 유지.

- ASYNC240 룰: async fn 안에서 `Path.exists()`/`is_file()`/`mkdir()` 등 호출 금지 — `asyncio.to_thread`로 래핑. fix 헬퍼: `_probe_path(p)` → `(exists, is_file)` 단일 stat 반환, `_rmtree_skill_storage(p)` → text(file)/package(dir) 분기 후 rmtree.
- `_derive_origin(item, user)` 매핑 (Spec §7.5):
  - is_system + 'k-skill' → built_in_k_skill
  - is_system + 'system_seed' → system_seed
  - !owner + restricted → shared_with_me
  - !owner + public → community
  - 그 외 (owner 본인 포함) → imported_by_me

## Codebase Patterns (젠슨 M5 Slice E, 2026-05-19)
- 보안 critical 패턴: sync 빌드 + async resolve 분리. `build_skill_runtime_context(cfg, *, data_dir)` (sync, copytree만) + `resolve_runtime_credentials(ctx, *, db, cfg)` (async, DB+decrypt+mutate ctx). 베조스 sync 테스트와 동시 호환.
- 두 ResolvedCredential 데이터클래스: `credential_requirements.ResolvedCredentialEntry` (leaf module — DB+cipher) → adapter → `skill_runtime.ResolvedCredential` (executor-facing). 단방향 import 유지.
- env_map 방향 (Spec/사티아 brief 기준): `{credential_field_name: env_var_name}`. 예: `{"username": "SRT_USERNAME", "password": "SRT_PASSWORD"}`. 따라서 `for field, env_name in rc.env_map.items(): env[env_name] = rc.decrypted[field]`.
- Override priority (Spec §8.4): `agent_skills.config.credential_bindings[key]` > `SkillCredentialBinding` row > missing → required면 fail-fast `marketplace_credential_required`. `scope='system_dependency' | 'manual'`은 Slice E 범위 밖.
- 4겹 보안 게이트: (1) 미선택 slug — descriptor lookup 실패 → reject. (2) 경로 traversal — `Path(input).name`으로 슬러그만 추출, `descriptor.runtime_storage_path.resolve()` + `is_relative_to(runtime_root)`. (3) 파일격리 — `copytree(symlinks=False)` 원본 쓰기 차단. (4) cleanup — mtime 기반 10분 interval / 1시간 retention.
- `build_skills_prompt`은 `/skills/<slug>/` 하드코딩 — executor에서 결과 텍스트에 `.replace("/skills/", f"/runtime/{thread_id}/skills/")` in-place rewrite. prompt.py public API 안 바꿈.
- Cipher V2 decrypt 경로: `credential_service.decrypt_with_external` (codebase 단일 진입점, __external__ ref expansion 포함). 실패 시 silent missing — required면 마지막에 묶어서 raise.
- redact_credential_values: 값 길이 정렬 (긴 것 먼저) 후 `text.replace(value, f"<redacted:<env_var>>")`. 5자 이상만 — 너무 짧으면 `id=x` 같은 정상 출력도 마스킹돼서 false-positive 폭증.
- redact_keys: `password|api_key|secret|token|access_key|refresh_token|client_secret|private_key` 패턴. 사이드카 key→`<redacted>` 치환만 (값 검사 X — 구조적 마스킹). dict/list/tuple recursive.
- streaming.py:TOOL_CALL_START parameters에 `redact_keys` 적용 (defense-in-depth) — 일반 도구도 auth 인자 받을 수 있어서.

## Codebase Patterns (젠슨 M6 Slice C + Task#15, 2026-05-19)
- secret_scan word boundary on `sk-` (Bezos OI-4): `\bsk-[A-Za-z0-9_\-]{20,}\b` — `sk-example`/`sk-foo` placeholder는 통과, 진짜 20+자 OpenAI key만 차단. False-positive 폭증 방지.
- secret_scan content scan은 256KB cap + binary extension skip — multi-GB 패키지가 reader OOM 안 일으킴.
- secret_scan `_iter_files`은 symlink 명시 skip — packager가 이미 reject하지만 defense-in-depth.
- secret_scan findings dedup: `(path, pattern)` 키로. 같은 파일에 같은 패턴 중복 매치는 1번만 보고, 다른 패턴은 별개로 (사용자에게 *왜* 차단됐는지 표시).
- Publish flow content-hash dedup: 동일 skill 재publish 시 기존 version 재사용 — 파일 시스템 + DB row 중복 회피. `tmp_target.rename(target)` 원자 이동 패턴.
- Publish flow는 `tmp_target = snapshot_target.with_suffix(".publish.tmp")` 경유. secret_scan fail 시 tmp만 cleanup, 기존 storage 안 건드림.
- Marketplace immutable version storage는 `data/skills/_marketplace_versions/<version_id>/`로 격리 — user-owned `data/skills/<skill_id>/`와 분리. 사용자 스킬 삭제가 installed version에 cascade 안 됨 (Spec §6.4).
- 라우터에서 `db.refresh(item)` 후 `project_item` 호출 시 lazy relationship (latest_version, acl_entries) 접근하면 `MissingGreenlet` 에러. 해결: `catalog_service.get_item`을 통해 재조회 (selectinload 적용됨).

- `publish_new_version` 라우터는 `PublishSkillIn` 풀 바디 합성 (기존 item.visibility/name/acl 등 + body.release_notes). 사용자는 item_id + release_notes만 보내고 서버가 나머지 채움.

## Codebase Patterns (젠슨 M7 Slice F, 2026-05-19)
- k-skill importer 분리 패턴: pure-function discovery/validation/hashing → DB-aware `import_skill` → orchestrator `run_sync`. 테스트가 각 레벨에서 독립 실행 가능.

## E2E 격리 DB + colima 포트 포워딩 gotcha (2026-06-12)
- 이 머신의 docker는 Docker Desktop이 아니라 **colima**다. 새로 publish한 컨테이너
  포트가 macOS 호스트로 포워딩되지 않을 수 있다 (VM 내부는 정상인데 호스트 nc만 실패).
  - 진단: `colima ssh -- bash -c 'echo > /dev/tcp/127.0.0.1/<PORT>'` 가 성공하는데
    호스트 `nc -z localhost <PORT>` 가 실패하면 colima 포워더 누락이다.
  - 복구: `colima ssh-config > /tmp/cfg && ssh -F /tmp/cfg -f -N -L <PORT>:127.0.0.1:<PORT> colima`
  - 정리: 포워딩은 colima의 SSH ControlMaster(mux)에 등록되므로 클라이언트 프로세스를
    죽여도 리스너가 남는다. `ssh -O cancel -L <PORT>:127.0.0.1:<PORT> -S ~/.colima/_lima/colima/ssh.sock lima-colima`
- live E2E용 격리 DB 레시피 (공유 DB에 alembic 금지 제약의 안전한 우회):
  1. `docker run -d --name moldy-e2e-pg -e POSTGRES_USER=moldy -e POSTGRES_PASSWORD=moldy -e POSTGRES_DB=moldy -p 5433:5432 postgres:16`
  2. worktree `backend/.env` 에 throwaway secrets 작성 (DATABASE_URL 5433,
     ENCRYPTION_KEY/JWT_SECRET 은 `openssl rand -hex 32`)
  3. `cd backend && uv run alembic upgrade head` (downgrade 왕복으로 신규 마이그레이션 대칭성도 검증 가능)
  4. `cd frontend && E2E_FRONTEND_PORT=3100 E2E_BACKEND_PORT=8101 pnpm playwright test e2e/smoke.spec.ts`
     (3000/8001/5432 는 다른 체크아웃·프로젝트가 점유 — reuseExistingServer 함정 방지용 대체 포트)
  5. 정리: `docker rm -f moldy-e2e-pg` + worktree `backend/.env` 삭제 + ssh 포워딩 cancel
- smoke.spec.ts 는 정기 스위트다. UI 구조 변경(사이드바 통합 등) 시 smoke 실패는
  "회귀 vs 낡은 단언"을 소스에서 판별한 뒤 테스트를 갱신한다. dropdown menuitem 은
  portal 렌더라 main 스코프가 아닌 page 레벨에서 찾는다.
- 사이드바/루트 레이아웃 컴포넌트는 ROOT_MESSAGE_NAMESPACES(appSettings, auth,
  common, metadata, nav, sidebar)만 클라이언트에서 쓸 수 있다
  (src/i18n/scoped-messages.tsx). chat.* 같은 페이지 스코프 네임스페이스를 참조하면
  런타임 IntlError — vitest 는 ko.json 전체를 주입하므로 못 잡고, E2E 의
  console-error 단언만 잡는다. 사이드바에 새 copy 추가 시 sidebar.* 아래에 둘 것.
