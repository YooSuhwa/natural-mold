## M1 (jensen) — 백엔드 opener_questions

### 구현 결과
- 마이그레이션: `m16_add_opener_questions` (m15 → m16). `agents.opener_questions JSON nullable`. 라운드트립 OK.
- 모델: `Agent.opener_questions: Mapped[list[str] | None]` (`JSON, nullable=True, default=list`) — `middleware_configs` 패턴 그대로.
- 스키마: `AgentCreate`/`AgentUpdate`/`AgentResponse` 3곳 모두 `opener_questions: list[str] | None`. 공유 validator `_validate_opener_questions`로 ≤12개 + strip 후 1~200자 검증. validator는 빈 문자열/공백 strip 후 reject(빈 항목 차단). 정상 항목은 strip된 값으로 정규화.
- 서비스: `create_agent`/`update_agent`에 한 줄씩 반영.
- 라우터: `_agent_to_response`에 `opener_questions` 필드 추가.
- 테스트: 4개 추가 (create+update+roundtrip / 13개 reject / 빈 문자열 reject / 201자 reject). 전체 pytest 628 PASS, ruff clean.

### 패턴/주의점
- **AgentCreate/Update는 `extra='forbid'`**: 새 필드 추가 시 두 클래스 모두에 등록 안 하면 422가 난다. 둘 다 등록 필수.
- **Pydantic v2 `field_validator` + `@classmethod`** 데코레이터 순서: `@field_validator(...)` 위, `@classmethod` 아래.
- **공유 validator**: 동일 검증 로직을 `Create`/`Update`에서 모두 쓸 때, 모듈 레벨 helper로 분리하고 두 클래스에서 호출하면 DRY.
- **라우터**: `_agent_to_response()`가 ORM → Response 변환의 단일 통로. 새 응답 필드 추가 시 여기 빠뜨리면 응답에 안 실린다.
- **PUT API**: 라우터는 `PUT /api/agents/{id}` (PATCH 아님). 테스트도 PUT 사용.
- **ruff I001**: 모듈 레벨 상수를 import 블록 직후에 추가하면 빈 줄 2개가 1개로 정리됨(`--fix`로 해결).
- **빈 리스트 `[]`로 PUT** 시 `is not None`이라 정상적으로 빈 리스트로 클리어됨. JSON 컬럼 default=list라 신규 row는 `[]`이 아닌 `None`(DB column nullable)이 될 수 있음 — 응답에서 `null` 그대로 노출.

### FE 계약
- `Agent.opener_questions: string[] | null` (저커버그 M6에서 동기화)
- `AgentUpdate { opener_questions?: string[] }` — undefined 보내면 변경 없음, `[]` 보내면 클리어, `null`은 PATCH로 클리어 가능(현재는 server-side validator가 `None`을 통과시키므로 무변경)

- VisualSettingsFlow `embedded + controlledState + controlledHandlers` 패턴이 hooks 규칙을 위반하지 않으면서 controlled/uncontrolled 모두 지원하는 깨끗한 해결책. 모든 read/write/useEffect/callback 경로에 `isControlled` 분기 적용. `embedded && !!controlledState && !!controlledHandlers` 명시적 가드로 불완전 prop 사고 차단.
- 워크벤치(`/agents/[id]/settings`)는 embedded=true 전달 → Toolbar 숨김 + page state owns. 기존 단독 라우트(`/agents/[id]/visual-settings`)는 embedded 미전달 → 기존 동작 100% 유지.
- TestChatPanel amber 배너로 MVP limitation 명시 — light/dark 모두 대응.

### 패턴 (재사용 가능)
- 같은 컴포넌트를 standalone vs embedded로 양쪽에서 쓸 때 `embedded?: boolean + controlledState? + controlledHandlers?` 트리오 + `isControlled = embedded && !!controlledState && !!controlledHandlers` 가드 + 모든 setter/effect 분기. internal state는 항상 useState로 호출(hooks 규칙) 하되 useEffect 안에서 `if (isControlled) return`으로 비활성화.
