## 프로젝트 목적

M3까지 PREBUILT 도구가 per-user default connection을 통해 credential을 해석한다. M4는 **CUSTOM 도구**도 `tool.connection_id` 경유로 이관한다. 현재 CUSTOM은 `tool.credential_id` 공유 행 참조 (PR #55 M3 완료 시점 기준 `_resolve_legacy_tool_auth` 경로).

**이 M4가 끝나면**: CUSTOM 도구 실행 시 `tool.connection.credential`로 복호화. 같은 credential을 공유하는 여러 CUSTOM tool은 1개 connection을 N:1로 공유. tool.credential_id / tool.auth_config는 M6 drop까지 legacy fallback.

## ADR-008 / exec-plan §4 M4 반영 규칙

- **CUSTOM Connection 형태**: `type='custom'`, `provider_name='custom_api_key'`, `extra_config=NULL`, `display_name=credential.name`, `is_default=true`, `status='active'`
- **N:1 공유 원칙**: 같은 credential을 참조하는 여러 CUSTOM tool은 동일 connection 공유 (m11에서 `(user_id, credential_id)` 단위 dedup)
- **Legacy fallback M6까지**: `tool.connection_id IS NULL AND tool.credential_id` 있으면 기존 `_resolve_legacy_tool_auth` 경로 — M6 cleanup에서 제거
- **Fail-closed 정책 (PREBUILT M3와 동일)**: `tool.connection_id` 지정된 상태에서 connection disabled / credential=NULL → `ToolConfigError` raise (env fallback으로 퇴행하지 않음)
- **Cross-tenant 가드**: `assert_connection_ownership` / `assert_credential_ownership` 재사용 (M2 도입, M3 확장)

## Alembic m11 상세 설계

### 대상 식별
```sql
SELECT id, user_id, credential_id FROM tools
WHERE type = 'custom'
  AND credential_id IS NOT NULL
  AND connection_id IS NULL;
```

### 이관 흐름 (idempotent)
1. 위 결과를 `(user_id, credential_id)` 단위로 distinct 그룹화.
2. 각 그룹에 대해:
   - 기존 `connections` 테이블에 `(user_id, type='custom', credential_id, display_name LIKE '[m11-auto-seed]%')` 존재하는가?
   - 있으면 **재사용** (conn_id 취함).
   - 없으면 **insert**:
     ```
     id = uuid4()
     user_id = tool.user_id
     type = 'custom'
     provider_name = 'custom_api_key'
     display_name = f'[m11-auto-seed] {credential.name}'
     credential_id = tool.credential_id
     extra_config = NULL
     status = 'active'
     is_default = true
     ```
3. 그룹 내 모든 tool의 `connection_id` 를 해당 conn_id로 UPDATE.

### downgrade
1. `display_name LIKE '[m11-auto-seed]%'` 조건의 connection을 대상으로 tool.connection_id = NULL로 역설정.
2. 해당 connection들 DELETE.
3. 수동 생성된 CUSTOM connection은 건드리지 않음.

### revision ID
- 파일명: `m11_custom_credential_migration.py`
- revision: `m11_custom_connection` (21자 — PG VARCHAR(32) 안전)
- down_revision: `m10_prebuilt_connection`

### 주의
- PostgreSQL 네이티브 SQL 사용 시 aiosqlite 테스트에서 실행 불가 → M9/M10 precedent처럼 helper 함수로 분리하고 pytest는 `inspect.getsource` 가드로만 검증. 실제 왕복은 PG에서 실행.
- `M11_SEED_MARKER = "[m11-auto-seed]"` 상수는 마이그레이션 파일 상단에 정의 후 display_name에 박음.

## 고위험 포인트

1. **credential 공유 → connection 1개** (N:1 shared). m11에서 `(user_id, credential_id)` dedup. 테스트로 강제.
2. **Legacy fallback 경로** — 기존 CUSTOM tool이 `connection_id IS NULL`로 남은 상태에서 실행 회귀 없어야 함. chat_service._resolve_legacy_tool_auth 경로 유지.
3. **Fail-closed 정책** — `connection_id` 있는데 disabled/credential=NULL → `ToolConfigError`. env fallback으로 회귀 금지 (PREBUILT는 env fallback이 있지만 CUSTOM은 env가 없어 어차피 동작 불가).
4. **ownership mismatch** — CUSTOM은 `tool.user_id NOT NULL` + `connection.user_id NOT NULL` + `credential.user_id NOT NULL` 3중 가드. DML 실수 방어.
5. **add-tool-dialog find-or-create UX** — credential 선택 → connection 자동 생성 흐름이 user-facing으로 보이지 않아야 함. 트랜잭션이 아니라 2-step (connection POST 성공 후 tool POST)이므로 실패 시 orphan connection 처리는 M5로 이월.
6. **M5 drive-by 금지** — custom-auth-dialog.tsx는 기존 tool의 credential re-bind에 쓰이지만 M4에서 교체하지 않음. Legacy 경로로만 동작.
