# progress.txt — feature/credentials-field-keys-cache (백로그 C)

팀 공유 메모리. 이전 작업의 progress는 `tasks/archive/progress-backlog-b.txt`로 아카이브됨.

## 작업 컨텍스트

`GET /api/credentials`가 각 row의 `data_encrypted`를 Fernet 복호화 + JSON 파싱해 `field_keys`를 노출 → N+1 복호화. `credentials.field_keys` 컬럼(비암호화 캐시)을 추가해 복호화 없이 키 이름만 반환.

**플랜 파일**: `~/.claude/plans/c-credentials-list-glistening-kurzweil.md`

## 사전 확인된 사실 (Phase 1 탐색 결과)

- `credential_service.list_credentials` (line 16-24): ORM only, 복호화 없음
- `credential_service.extract_field_keys` (line 98-103): `resolve_credential_data()` → `decrypt_api_key()` 호출 → N+1 원인
- `credential_service.create_credential` (line 42-66): line 55 암호화, 여기서 `field_keys` 캐시도 저장해야 함
- `credential_service.update_credential` (line 69-82): `data.data is not None` 분기에서 동기화
- 라우터 `_to_response()` (credentials.py:23-35): 응답 스키마는 변경 불필요
- 응답 스키마 `CredentialResponse` (schemas/credential.py:21-32): `field_keys: list[str]` 필드 그대로
- Alembic head: `m6_add_credentials` (backend/alembic/versions/m6_add_credentials.py)
- `sa.JSON()`이 프로젝트 관례 (agent_tools.config, builder_sessions.*, models.input_modalities 등)
- ENCRYPTION_KEY는 `.env`에 설정됨 — backfill 마이그레이션에서 `app.services.encryption.decrypt_api_key` 사용 가능

## 결정된 설계 (사용자 confirm)

| 항목 | 결정 |
|---|---|
| 컬럼 이름 | `field_keys` (모델, 응답 스키마 동일명 — 자연스러움) |
| 컬럼 타입 | `sa.JSON()` nullable=True (PostgreSQL JSONB 매핑, SQLite 호환) |
| Backfill 전략 | **마이그레이션 upgrade() + runtime fallback 병행** (사용자 선택) |
| extract_field_keys | 캐시 우선, NULL이면 복호화 fallback (legacy 대응) |
| 테스트 | 신규 `test_credentials.py` 신설 (기존 분산됨) |

## Gotchas (현재까지)

- 데코레이터 `sa.JSON()`은 PostgreSQL에서 JSONB로 매핑되지만 명시적 JSONB 사용 금지(프로젝트 관례)
- 마이그레이션 내 `from app.services.encryption import decrypt_api_key` 가능하나, 실패 시 `[]`로 tolerant 처리
- aiosqlite 테스트 환경: 백필 마이그레이션이 돌려지더라도 빈 테이블 → 무영향
- 응답 스키마 불변 (API 소비자 호환성)
- ENCRYPTION_KEY 미설정 환경에서 마이그레이션은 스킵 (503 반환 환경이므로 create도 거부됨)
