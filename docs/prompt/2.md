## 📋 Tool Guidelines 작성 가이드

### 기본 구조

각 도구마다 다음 **3가지 필수 요소**를 포함해야 합니다:

```markdown
### `{도구명}`
- Purpose: [기능 설명]
- When: [언제 사용할지]
- Caution: [주의사항]
```

---

## 🎯 각 요소별 작성 방법

### 1️⃣ **Purpose (목적/기능)**

**무엇을 하는 도구인지** 간단명료하게 설명

#### ✅ 좋은 예시
```
- Purpose: 인터넷에서 최신 정보 검색
- Purpose: 사용자 질문을 분석하여 명확화
- Purpose: 파일 내용을 읽고 처리
```

#### ❌ 나쁜 예시
```
- Purpose: 도구 (너무 모호함)
- Purpose: 여러 가지 일을 할 수 있음 (구체성 부족)
```

#### 💡 작성 팁
- 1-2 문장으로 간결하게
- 도구가 **직접 수행하는 행동** 중심으로 작성
- 기술 용어보다는 **사용자 관점**에서 설명

---

### 2️⃣ **When (사용 시기/트리거)**

**어떤 상황에서 이 도구를 사용해야 하는지** 명시

#### ✅ 좋은 예시
```
- When: 
  - 사용자가 특정 주제에 대한 정보를 요청할 때
  - 최신 뉴스나 트렌드 정보가 필요할 때
  - 사실 확인이 필요할 때
```

```
- When:
  - 사용자의 요청이 모호하거나 불명확할 때
  - 여러 해석이 가능한 질문을 받았을 때
  - 추가 정보가 필요한 경우
```

#### ❌ 나쁜 예시
```
- When: 필요할 때 (너무 모호함)
- When: 항상 (구체성 부족)
```

#### 💡 작성 팁
- **구체적인 상황/조건** 나열
- 불릿 포인트로 여러 경우 제시
- "~할 때", "~가 필요할 때" 형식 사용
- 도구를 **언제 호출할지** 판단 기준 제공

---

### 3️⃣ **Caution (주의사항)**

**도구 사용 시 피해야 할 것, 주의할 점** 명시

#### ✅ 좋은 예시
```
- Caution:
  - 검색 쿼리는 명확하고 간결하게 (400자 이내)
  - 신뢰할 수 있는 출처의 정보를 우선시
  - 검색 결과가 없으면 다시 시도하기
```

```
- Caution:
  - 사용자의 개인정보를 요청하지 않기
  - 민감한 정보는 암호화하여 처리
  - 파일 크기 제한 확인 (최대 10MB)
```

#### ❌ 나쁜 예시
```
- Caution: 조심해서 사용하기 (너무 모호함)
- Caution: 실수하지 말기 (구체성 부족)
```

#### 💡 작성 팁
- **구체적인 제약사항** 나열
- "~하지 말기", "~에 주의", "~를 확인" 형식
- 도구의 **한계나 제한사항** 명시
- 성능/비용 관련 주의사항 포함 (필요시)
- 보안/개인정보 관련 주의사항 포함 (필요시)

## 📊 도구 유형별 작성 예시

### 🔍 검색 도구 (tavily_search, naver_webkr 등)

```markdown
### `tavily_search`
- Purpose: 인터넷에서 최신 정보와 뉴스 검색
- When:
  - 사용자가 특정 주제에 대한 정보를 요청할 때
  - 최신 뉴스, 트렌드, 사실 확인이 필요할 때
  - 일반적인 지식 질문에 답변할 때
- Caution:
  - 검색 쿼리는 명확하고 간결하게 (400자 이내)
  - 필요에 따라 여러 번 검색하여 정보 확인
  - 신뢰할 수 있는 출처의 정보를 우선시
  - 검색 결과가 없으면 다른 검색어로 재시도
```

### 🤔 명확화 도구 (ask_clarifying_question)

```markdown
### `ask_clarifying_question`
- Purpose: 사용자의 모호한 요청을 명확히 하기 위해 질문하기
- When:
  - 사용자의 요청이 여러 해석이 가능할 때
  - 필요한 정보가 부족할 때
  - 사용자의 의도를 정확히 파악해야 할 때
- Caution:
  - 한 번에 하나의 질문만 하기 (여러 질문 금지)
  - 3개의 선택지와 자유 입력 옵션 제공
  - 사용자의 답변을 기다린 후 다음 단계 진행
  - 이미 명확한 요청에는 사용하지 않기
```

### 📁 파일 처리 도구 (read_agent_file, list_permanent_files)

```markdown
### `read_agent_file`
- Purpose: 업로드된 파일의 내용을 읽고 처리하기
- When:
  - 사용자가 파일 기반 질문을 할 때
  - 문서의 특정 부분을 찾아야 할 때
  - 파일 내용을 분석하거나 요약해야 할 때
- Caution:
  - 파일 ID를 정확히 확인한 후 읽기
  - 대용량 파일는 일부만 읽을 수 있음
  - PDF는 마크다운으로 변환되므로 포맷 변화 고려
  - 파일 내용을 직접 프롬프트에 복사하지 않기
```

### 🔧 설정/구성 도구 (update_system_prompt, update_model_config)

```markdown
### `update_system_prompt`
- Purpose: 에이전트의 시스템 프롬프트 전체 업데이트
- When:
  - 새 에이전트를 처음 설정할 때
  - 프롬프트를 완전히 다시 작성해야 할 때
  - 기존 프롬프트가 없을 때
- Caution:
  - 기존 지침을 임의로 삭제하지 않기
  - 도구/미들웨어 변경 시 함께 업데이트하기
  - 변경 후 에이전트 동작 확인하기
  - 부분 수정은 edit_system_prompt 사용 권장
```

### 🔄 반복 작업 도구 (create_cron_schedule)

```markdown
### `create_cron_schedule`
- Purpose: 에이전트의 자동 실행 일정 설정
- When:
  - 정기적으로 반복 실행해야 할 때 (매일, 매주 등)
  - 특정 시간에 한 번만 실행해야 할 때
  - 자동화된 작업이 필요할 때
- Caution:
  - Cron 표현식 형식 정확히 확인 (5필드)
  - 타임존 설정 확인 (기본값: Asia/Seoul)
  - 최대 20개 스케줄까지만 생성 가능
  - 일회성 스케줄은 미래 시간으로 설정
```

## 🎨 작성 스타일 가이드

### 형식
```markdown
### `도구명`
- Purpose: [설명]
- When: [상황 1], [상황 2], [상황 3]
- Caution: [주의사항 1], [주의사항 2]
```

또는 복잡한 경우:
```markdown
### `도구명`
- Purpose: [설명]
- When:
  - [상황 1]
  - [상황 2]
  - [상황 3]
- Caution:
  - [주의사항 1]
  - [주의사항 2]
  - [주의사항 3]
```

### 언어
- 한국어로 자연스럽게 작성
- 존댓말 사용 (에이전트 지침이므로)
- 명령조 사용 ("~하기", "~하지 않기")

### 길이
- Purpose: 1-2 문장
- When: 2-4개 항목
- Caution: 2-4개 항목

## ✅ 체크리스트

Tool Guidelines를 작성할 때 다음을 확인하세요:

- [ ] **Purpose**: 도구가 무엇을 하는지 명확한가?
- [ ] **When**: 언제 사용할지 구체적인 상황이 나열되어 있는가?
- [ ] **Caution**: 주의할 점이 명확하고 실용적인가?
- [ ] **구체성**: 모호한 표현이 없는가?
- [ ] **도구 특성**: 도구의 특징과 한계가 반영되어 있는가?
- [ ] **사용자 관점**: 에이전트가 이해하고 따를 수 있는가?

---

## 💡 실전 팁

### 1. 도구 문서 먼저 읽기
- 도구의 파라미터, 반환값, 제한사항 확인
- 도구의 실제 기능과 한계 파악

### 2. 에이전트 관점에서 작성
- "에이전트가 언제 이 도구를 써야 할까?"
- "이 도구를 잘못 사용하면 어떻게 될까?"

### 3. 구체적인 예시 추가 (선택)
- 복잡한 도구는 사용 예시 추가 가능
- 예: "검색 쿼리 예시: '파이썬 최신 버전 기능'"

### 4. 도구 간 관계 명시 (필요시)
- 여러 도구를 함께 사용할 때 순서 명시
- 예: "먼저 list_permanent_files로 파일 확인 후 read_agent_file 사용"
